# Dense progress capture design

Round 22 registers the documented CPLEX 22.1.1 generic callback in `GLOBAL_PROGRESS` and `LOCAL_PROGRESS` contexts. CPLEX deliberately calls global progress sparsely and did not emit local progress for the Tailored callback mask in the pre-refreeze live gate, despite thousands of processed nodes. Tailored therefore also takes the same callback-safe information snapshot at the start of its already-required documented `RELAXATION` context, before any algorithmic row or branch action. The capture subpath is read-only: it cannot add rows, branch, reject a candidate, alter estimates, abort, change parameters, or invoke an optimizer. Official solves use one CPLEX thread, avoiding concurrent mutation of the recorder. Existing candidate, relaxation, and branching algorithm behavior is unchanged.

The local context supplies genuinely dense in-tree observations for plain CPLEX; the relaxation snapshot supplies them for Tailored when global/local progress is sparse. Callback-safe values are queried with `CPXcallbackgetinfo*`: elapsed/deterministic time, best bound, incumbent, processed/open nodes, and simplex iterations. Unsupported solution-count and tree-memory observations remain unavailable. Tailored metadata supplies Gini/ordinary branch counts and sibling-delay state under its existing mutex. Plain and Tailored use the same `DenseProgressRecorder`, retention rules, CSV schema, checkpoint grid, freshness calculation, and integrity audit.

The recorder retains the first observation, all bound/incumbent/phase/branch/solution changes, material node or open-node changes (100 by default), and scheduled heartbeats. Heartbeat cadence is 1 second before 60 seconds, 5 seconds before 300, 10 seconds before 900, and 30 seconds thereafter. If CPLEX provides no callback-safe observation in a cadence window, extraction marks the checkpoint stale or not observed; it never manufactures a row. Equal clock readings are advanced by one representable `double` to keep timestamps strict.

Records are buffered in memory with a 500,000-record bound and written once after native finalization. A full-precision `solver_final_native_api` event is forced after `CPXmipopt` evidence capture. Invocation count, retained/dropped count, callback wall time, serialization time, peak buffer bytes, final-record state, flush state, and instrumentation percentage are serialized. Logging-off diagnostics are separated from official performance rows.
